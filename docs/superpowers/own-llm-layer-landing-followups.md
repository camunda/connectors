# Own-the-LLM-layer (#7211) — follow-ups to tackle when this lands

Tracked gaps that are acceptable for the pilot but **must be revisited before / as this ships**.
Grouped by whether they need an engine-side change or are connector-only.

## Engine-side (need a schema / API change in the engine + camunda-client-java) — HIGHEST PRIORITY when landing

These are the ones flagged explicitly (Mathias, 2026-07-14). They cannot be fixed connector-side alone.

1. **First-class `AgentInstanceHistoryContent` types for `ProviderContent` and `ReasoningContent`.**
   - Today `AgentInstanceHistoryMapper.toHistoryContent` degrades BOTH `ProviderContent` (server-tool / code-execution / container blocks) and `ReasoningContent` (thinking) to `AgentInstanceHistoryContent.object(payload)` — the engine client's content vocabulary is only `text` / `object` / `document` (camunda-client-java 8.10.0-SNAPSHOT).
   - Consequence: on the engine side these round-trip as opaque JSON objects but are **not semantically rendered**, and server tool calls never appear in the instance's typed `toolCalls`.
   - Fix requires an **engine-side schema addition** (new `AgentInstanceHistoryContent` variants) + the mapper branches to emit them. Same class of gap as item 2.

2. **`systemPrompt` on `UpdateAgentInstance`.**
   - The engine's `UpdateAgentInstance` does not accept a `systemPrompt`, which blocks the "compose at READY, push to instance" approach (Option 1) for the agent-instance system-prompt gap.
   - Needs the engine to accept `systemPrompt` on the update path.

## Connector-side (can be done in this codebase; deferred by scope, not blocked)

Declared-but-unused: `supportsReasoning`, `supportsReasoningSignatureRoundtrip`, `supportsParallelToolCalls`, `supportsPromptCaching`, `assistantMessageModalities`, `contextWindow`. Decision pending: wire the cheap ones (`supportsParallelToolCalls` → `disable_parallel_tool_use`) vs trim unused fields from the matrix vs document-as-declared.

4. **~~Reasoning round-trip is one-directional.~~ DONE (verified 2026-07-16).** The R1 reasoning+effort chunk closed this. `AnthropicMessageResponseConverter` now captures the full raw thinking (and `redacted_thinking`) block — including its `signature` — as `ReasoningContent.providerPayload`, and `AnthropicContentConverter` (`case ReasoningContent`) replays it byte-identically as a `BetaContentBlockParam`, positioned before `tool_use`. The native path ALWAYS populates `providerPayload`, so the old "empty content array" risk is only reachable for a **null-payload** `ReasoningContent`, which only the L4J bridge produces — a non-issue for the native pilot. Covered e2e by `NativeAnthropicReasoningEffortIT.roundTripsSignedThinkingBlockBeforeToolResultOnFollowUpRequest` (WireMock, real accumulator). Reasoning config/UX polish still tracked under #7669.
   - Original NEXT-CHUNK note (kept for history): **(Mathias, 2026-07-14): do reasoning next**, dragging reasoning config into the connector now to get the data structures right. Ratified design direction: **provider-specific request config, provider-neutral response data.** Config is provider-typed (like the rest of the wire-format-first pilot): Anthropic gets `thinking {enabled, budget_tokens}` on `AnthropicConnection` (capability-gated by `supportsReasoning` — its first real consumer); OpenAI later gets `reasoning_effort`. Do NOT force a neutral reasoning-config abstraction. The neutral layer is the response-side `ReasoningContent` (text + signature/redacted payload), which all providers fold into. **Config-shape spectrum to accommodate — the neutral model + config must handle all three: token BUDGET (Anthropic classic, Gemini), reasoning-EFFORT LEVEL (OpenAI low/med/high, GPT-5 incl. `minimal`), and ADAPTIVE THINKING / effort (newer models expose effort-style / adaptive control — do NOT model reasoning as budget-only).** Open design decisions to pin in a short brainstorm→spec first: signature/redacted placement (typed field vs `providerPayload`); interleaved-thinking (beta) interaction with tool calls; exact `ReasoningContent` shape for lossless multi-turn round-trip.

5. **~~Prompt caching — not wired.~~ WIRED (opt-in, verified 2026-07-16).** `AnthropicChatModel.enablePromptCaching` (`configuration.anthropic.enablePromptCaching`, default `false`) → `AnthropicMessageRequestConverter.applyPromptCaching` emits a **top-level** `cache_control: {ephemeral}` (Anthropic automatic caching; breakpoint placement delegated to the API). Verified real-API: both Anthropic acceptance rows assert cache-creation + cache-read tokens > 0. Remaining, tracked (not blockers): (a) it is top-level automatic, NOT curated per-block breakpoints on system/tools/history (self-labeled a "spike"); (b) the sliding `MessageWindowFilter` busts the *message-history* portion of the cached prefix once it starts evicting — system+tools stay cached (only bites past the window threshold); (c) there is **no deterministic (WireMock) assertion of `cache_control` on the request** — only the real-API token-count assertion. Caching config/UX polish still under #7668.

6. **Web-tools dynamic filtering.** Default web-tool revisions (`web_search_20250305` / `web_fetch_20250910`) call directly and coexist with Skills — this works. Only the case of a user OVERRIDING to a dynamic-filtering revision (`web_search_20260209`+ / `web_fetch_20260209`+) alongside Skills collides on `code_execution` (400). `allowed_callers` management is deferred; consider fail-fast connector-side validation instead of letting the API 400 through.

7. **Conversation variable-space bloat (measured).** VERIFIED against a live local PI (c8ctl `search vars --name agentContext --fullValue`): the in-process store inlines the ENTIRE conversation as JSON into the `agentContext` process variable — **518 KB** in a modest run — dominated by Anthropic-returned data stored byte-identical: `server_tool_use` / `code_execution` result blocks (`ProviderContent`), plus `thinking` blocks + signatures. No `encryptedContent` field / no document reference — it's raw inline JSON. Grows unbounded with turns + server-tool activity; risks Zeebe variable-size limits + perf. Because prompt caching (item 5) will need this data byte-identical, we must NOT prune it. The right lever is to OFFLOAD it out of the process variable: `CamundaDocumentConversationStore` (already exists) backed by an EXTERNAL document store, or an external conversation-store backend — variable holds a pointer, bytes live out-of-band, still byte-identical retrievable. CAVEAT: the in-MEMORY document store re-embeds the (base64, ~+33%) content back into the variable, so documents only help with a real external store. Follow-up: recommend/wire an external store for cache-heavy / long-conversation workloads.

8. **Untested Anthropic branches.** `mapStopReason` STOP_SEQUENCE / REFUSAL / default, and the API-level `Continuation` path (C7 review N2). `redacted_thinking` is production-handled and being covered by a WireMock round-trip test in the 2026-07-16 acceptance-parity chunk (item 13).

10. **Call watchdog (deadline abort) — DEFERRED past the pilot (Mathias, 2026-07-14).** `AnthropicChatModelApi.call()` blocks in `streamAssembler.assemble(stream)`. The only timeout wired is the SDK client's `builder.timeout(timeout)` (`AnthropicOkHttpClientFactory:49`), which for a *streaming* call degrades to a *per-read* (inter-event) timeout — there is **no total wall-clock bound** on a stream that keeps emitting events (long server-tool runs, `pause_turn` continuations, slow-but-alive token streams). A live-but-too-long call pins the connector thread until the Zeebe job times out (→ retry while the old thread still runs → double-execution risk). Design when we pick this up: one shared daemon `ScheduledExecutorService` (Spring bean, NOT a thread per call); on each streaming call schedule a task at the deadline that sets an `AtomicBoolean timedOut` and calls `stream.close()` (StreamResponse is AutoCloseable — cross-thread close is the SDK's documented cancellation path and unblocks the reading thread's `assemble()`; `Thread.interrupt()` would NOT reliably unblock a socket read); cancel the guard on normal exit; translate a `timedOut` abort to a NEW `ERROR_CODE_MODEL_CALL_TIMEOUT` (distinct from `ERROR_CODE_FAILED_MODEL_CALL`). Deadline source = reuse `connection.timeouts().timeout()` (keep the SDK per-read timeout as-is; watchdog owns the total-call bound). Ship per-round first; the seam generalizes to a per-turn budget across `pause_turn` continuations via `min(perRoundDeadline, turnBudgetRemaining)`. VERIFY FIRST: confirm anthropic-java drops the total-request timeout for streaming (read-timeout-only) — if it actually honors a total request timeout on streams, the watchdog's HTTP-abort value shrinks to just the turn-budget/cancellation seam. Kept SYNC SPI — the hackdays streaming+listener surface only pays off once an async handler chain consumes it; not needed for abort.

11. **Raw-wire TRACE logging (native providers) — DEFERRED, delegate (Mathias, 2026-07-16).** The DEBUG "assembled object" layer (SDK request-params + assembled response, serialized via the SDK `ObjectMappers.jsonMapper()`, on `…framework.openai`/`…framework.anthropic`, once per call, streaming-safe) is SHIPPING now. The lower raw-wire layer is deferred: HTTP request method/URL + **redacted** headers (`authorization`/`x-api-key`/`api-key`/proxy-auth) + request body, and response status + redacted headers, on a dedicated child logger `…framework.<provider>.wire` at **TRACE**, so Spring Boot logger inheritance cascades (`…framework.openai=TRACE` ⇒ both DEBUG assembled + TRACE wire). Design (already researched, 2026-07-16): install a per-SDK `com.<vendor>.core.http.HttpClient` **decorator** that delegates to the SDK transport and logs around `execute()`, injected via `client.withOptions(b -> b.httpClient(decorator))` (both SDKs expose `ClientOptions.Builder.httpClient(...)` + `withOptions` + public `OkHttpClient(okhttp3.OkHttpClient)` ctor and `OkHttpClient.builder()`). Install the decorator ONLY when the `.wire` logger `isTraceEnabled()` at client-build time (clients are built per call, so runtime level changes are picked up; zero overhead + zero behavior change when off). **Response BODY is intentionally NOT re-logged raw** — we always `createStreaming()`, and teeing the SSE stream (the SDK's own `LoggingInputStream` approach, which otherwise writes to `System.err`) is the fragile part we're avoiding; the response body content is already visible in the DEBUG assembled-response object. Per-SDK decorators required (distinct `com.openai.core.http.*` vs `com.anthropic.core.http.*` packages). NOTE: the SDKs' built-in `ANTHROPIC_LOG`/`OPENAI_LOG` env-var logger writes to `System.err` (bypasses SLF4J/Logback) — not the supported path. Doc stub already in `ai-agent.md` §12 "Request/response debug logging (native providers)".
